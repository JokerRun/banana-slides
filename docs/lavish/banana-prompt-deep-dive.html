一屏结论：真正目标是“去硬编码、让 Preset/用户输入接管风格”

P0
Generate PPT generic prompt 需要瘦身
generic builder 不应包含 DDI palette、咨询风、固定标题/图形规范；只保留任务编排、内容忠实、引用图片规则。
P0
DDI 规则只从 DDI Preset 来
选择 DDI 时最大化使用 DDI preset；未选择 DDI 时，不得工程层硬套 DDI 规范。
P1
Edit 与 Restyle 同源
普通 Edit 不应继续一套独立小 prompt；应复用 Restyle-like 图到图 context：baseline snapshot + current image + edit instruction + refs。
核心纠偏：上一版报告只把问题描述成“generic prompt + DDI prompt 冗余”。更准确的产品/工程目标是：generic generation 本身就不该硬编码任何 DDI 或固定咨询风规范；风格应该来自 preset 或用户提供的参考图/文字。DDI 只是一个 preset，不是系统默认设计宪法。

批注后修订方案：三条主线，一步到位

主线 A：Generate Prompt 去硬编码

P0

把 get_image_generation_prompt() 从“资深商业咨询级 PPT 设计师完整 prompt”降级为中立 orchestrator。

  • 保留：内容不能改、参考图引用、layout hint 使用方式、输出 16:9。
  • 移除：DDI palette、固定咨询风、硬编码标题格式、硬编码图形/图标补偿规则。
  • 风格：只来自 style_preset_id / 用户 style refs / 用户 prompt。

主线 B：Preset Contract 一源化

P0

DDI 规则留在 assets/presets/ddi/；选择 DDI 时完整使用 DDI preset，未选择时完全不注入 DDI。

  • DDI selected：base image + DDI prompt + DDI metadata。
  • Custom style：custom refs + custom style text。
  • No style：只根据用户内容生成，不套默认 DDI。

主线 C：Edit 与 Restyle 同源

P1

普通 Edit、Restyle Edit、Translate Restyle 后续编辑都走同一个 I2I context builder，而不是各自散落 prompt。

  • baseline：原始生成 prompt snapshot + baseline refs。
  • current：当前图 + 当前版本 refs。
  • delta：用户 edit instruction。

冗余高发点：新 PPT 生图 prompt 的实际骨架

当前文本转图 prompt 结构

# Role: 资深商业咨询级 PPT 内容架构师与视觉设计师 # Inputs: 参考图片 / [文本] / ASCII Layout # Core Objective: 基于文本 + 套用参考图风格，从零设计咨询风 PPT <page_description> 页面标题 / 页面文字 / 布局建议 ASCII / 图片 markdown </page_description> <reference_information> 整个 PPT 大纲 + 当前章节 </reference_information> <execution_rules> 文字清晰、16:9、深度解析参考图 禁止改写文本 Layout 只作为指令，不渲染 ASCII 1:1 文本条目映射 标题规范、DDI palette、版式选择指南 图形类型、留白、文字/图形比例、图标补偿 </execution_rules> 额外要求（请务必遵循）： # Role: 资深商业咨询级 PPT 排版与视觉架构师 # Inputs: [参考图片] / [文本内容] # Core Objective: 基于参考图片作为 Master Template... ## 模板迁移规则 ## 零重写内容原则 ## 内容逻辑理解与结构重组 ## 标题格式与定位规范 ## 色系规范 ## 商业视觉风格 ## 智能视觉补偿 # Output Format: 16:9 高保真商业 PPT 页面

问题本质

1
Role 重复
两个“资深 PPT 设计师/架构师”角色，模型不清楚哪个是主系统。
2
目标重复
两个 Core Objective 都要求咨询风、模板迁移、重构信息架构。
3
约束重复
零改写、色彩、标题、图形、版式、输出格式都重复。
4
信号优先级混乱
page_desc 的 ASCII layout、generic layout guide、DDI prompt layout guide、reference image 同时指导版式。
为什么会影响质量：图像模型对长 prompt 并不等价于“更严格”。重复规则会稀释页面内容本身和模板图像的权重，尤其在 layout / color / visual compensation 上，模型更容易过度设计、模板遵从性下降、文案区域分配失真。

维护性与 DSPy 评估：现在更缺 snapshot，不是 optimizer

当前 prompt 维护问题

问题表现建议
Prompt source 多层叠加通用 builder、canonical preset markdown、page description、layout suggestion 都能给 layout/style 指令。先定义 prompt layer contract：content / layout / style / refs / output，不允许跨层重复。
缺少最终 prompt snapshot改了 preset 或 builder 后，不容易看出 provider 实际收到什么。补 snapshot harness：输入 fixture project，输出 final_prompt + ref_image_manifest。
Preset markdown 太像完整 promptDDI generate prompt 包含 role/objective/output，和 builder 冲突。把 generate preset 收敛成 compact style contract；完整 prompt 只用于 Restyle/Translate 专用场景。
真实质量评估缺闭环只能靠人工看图，定位慢。先人工 golden set，再考虑半自动评分。

DSPy 是否适合？

用途适合度理由
直接修当前冗余 prompt不推荐这是确定性的 prompt assembly 问题，用工程拆层和 snapshot 更直接。
生成 prompt variants 并优化中期可用需要先有可靠 metric/golden set，否则 optimizer 会优化到错误目标。
评估 layout/template compliance可探索可用 VLM judge + rubric，但要先标注少量真实样例。
生产时动态拼 prompt不建议生产链路应 deterministic、可审计、可复现；DSPy 更适合离线优化。

推荐的可维护实现形态（伪代码）

class ImagePromptAssembler: def build_generate_prompt(project, page, refs): content = ContentLayer.from_page_description(page.description) layout = LayoutLayer.from_page_description(page.description.layout_hint) style = StyleLayer.from_preset(project.style_preset_id) refs = RefLayer.from_project_refs(project, page) # Rule: each layer owns exactly one concern. # style layer must not define role/objective/output format. # layout layer must not restate color/template rules. return render_template( role="PPT image generator", content=content.required_text, layout=layout.compact_hint, style=style.compact_contract, refs=refs.manifest, output="16:9 high-fidelity slide image", ) def snapshot_prompt(project_fixture): prompt, refs = ImagePromptAssembler.build_generate_prompt(...) assert_no_duplicate_sections(prompt, ["Role", "Core Objective", "Output Format"]) assert_ref_manifest_matches_images(refs)
结论：先不要引入 DSPy 进生产路径。先做 deterministic prompt assembler + snapshot；等有 golden set 和 VLM rubric 后，再用 DSPy/GEPA 离线探索更优 prompt wording。

待确认问题

问题为什么重要建议验证方式
无 preset 的默认风格应有多“空”？如果完全不加风格，模型可能自由发挥；但这比隐藏 DDI bias 更符合产品原则。用 no-style fixture 看质量；只允许加入中立质量约束，不允许固定品牌/风格。
DDI generate preset 是否保留完整 prompt？选择 DDI 时应该最大化使用 DDI preset；但仍需避免和 neutral orchestrator 重复 role/output。DDI preset 可保留风格细节，neutral orchestrator 不再定义风格。
Edit 同源化 metadata 放哪里？普通 Edit 需要 baseline prompt/ref manifest；现有数据结构可能不足。优先挂在 PageImageVersion metadata；没有字段则加 JSON 字段或复用现有 version metadata。
